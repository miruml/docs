## Welcome

Welcome\! Miru is an application configuration management solution for robotics teams. We provide tools for engineers to define, deploy, and manage application configurations for their robots.

## Difficulties of Configurations

Robots rely on application configurations stemming from a variety of sources. Some common types include:

- **Calibrations**: Each end effector, gripper, and sensor requires unique calibration settings.
- **Hardware Variability**: Hardware variants use configurations to set the appropriate software dependencies and functionalities the hardware supports.
- **User Preferences**: End users desire different settings or features.
- **Testing & Development**: Test beds and beta robots need feature flagging and experimentation.

These configurations touch many different engineers on the team, each in their own way:

- **Software Engineers**: Depend on configs to run code reliably across robots in the fleet.
- **Hardware Engineers**: Must safely push config changes to robots to resolve issues and limit downtime.
- **End Users**: Frequently tweak config parameters during acceptance testing.
- **Field Service Technicians**: Adjust configs on robots in the field during calibration or troubleshooting.


### Editing Configurations with SSH

Managing this complexity is often left to ad‑hoc tools. Often, Deployment Engineers must SSH into robots and edit text files directly with Vim:
- This workflow is insecure, error‑prone, and impossible to scale.
- It leads to downtime, slow fixes, and forces teams to hire software‑savvy Deployment Engineers, even when the preferred background is Mechanical or Electrical Engineering.

The brittle nature of this approach leads to a bigger problem when teams lose visiblity into the configuration state of their fleet.

### Underlying Complexity

The challenges above are compounded by the natural complexity of robotics' configurations:

- **Complex Groupings**: Robots belong to overlapping groups by customer, hardware variant, and release channel. Defining and updating the right config is hard.

- **Local edits**: Edits happen locally on the robot, and thanks to poor networking and firewalls, are not always reflected in the cloud.

- **Configuration volume**: A single robotic system can have a thousand configuration parameters!
